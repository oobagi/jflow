# Docs

Sync project documentation with the actual state of the codebase. Fix stale references surgically — never rewrite the author's voice, tone, or structure. Only change factual content that is demonstrably wrong or outdated.

## 1. Inventory existing docs (run in parallel)

Read all documentation files and project metadata in parallel:

**Documentation files:**
- `README.md` — project description, features, quickstart, shields
- `ROADMAP.md` — phases, items, issue links, checkbox state
- `AGENTS.md` — project context, tech stack, build/run/test commands, structure
- `CHANGELOG.md` — version history (if it exists)
- `docs/` — any files in the docs directory
- `DESIGN.md` — design system (if it exists)
- `todo.txt` — manual setup checklist generated by /setup (untracked, in .gitignore)
- `LICENSE` — license text (if it exists)

**Project metadata (source of truth):**
- Package manifest: `package.json`, `Cargo.toml`, `pyproject.toml`, `go.mod`, `Package.swift` — for dependency versions, project name, description
- Project structure: `find . -not -path './.git/*' -not -path './node_modules/*' -not -path './target/*' -type f | head -60` — for verifying file path references
- Git log since docs last changed: for each doc file, run `git log --oneline -1 --format='%H %cr' -- <file>` to get when it was last updated, then `git log --oneline <hash>..HEAD` to see what changed since
- GitHub issues: `gh issue list --state all --limit 50` — for verifying ROADMAP.md checkbox state

## 2. Detect staleness

For each doc file, check for these categories of staleness:

### File path references
- Grep each doc for paths like `src/`, `lib/`, `docs/`, `assets/` etc.
- Verify each referenced path actually exists
- Flag any that point to moved or deleted files

### Dependency versions
- Compare versions mentioned in docs against the actual package manifest
- Common patterns: shield badges with version numbers, install instructions with specific versions, compatibility tables

### Feature lists
- Compare features listed in README against what's actually implemented
- Check if new modules/commands/endpoints exist that aren't documented
- Check if documented features have been removed

### ROADMAP.md checkboxes
- For each roadmap item with a `#N` issue link, check the issue's current state via `gh issue view N --json state --jq .state`
- Items linking to closed issues should be `[x]`, open issues should be `[ ]`
- Flag any mismatches

### Build/run/test commands
- Compare commands in README quickstart and AGENTS.md against actual scripts in the package manifest
- Check if `npm test`, `cargo test`, `pytest`, etc. still work as documented

### Project structure descriptions
- If AGENTS.md or README describes the directory structure, verify it matches reality
- Check for new top-level directories not mentioned in docs

### todo.txt (if it exists)
`todo.txt` is an untracked personal checklist generated by `/setup`. It's not a doc to sync — it's a doc to **prune**. Check each unchecked item to see if it's already been done:
- File references: if an item says "drop icon at `assets/foo-icon.png`" and that file now exists, check the box
- Environment/secrets: if an item references a secret like `ROADMAP_PAT`, check if it's set via `gh secret list` or `echo "${VAR:-(not set)}"`
- Service setup: if an item says "set up Supabase" and relevant config (env vars, `supabase/` dir, etc.) already exists, check the box
- If all items are checked, suggest the user delete the file

### LICENSE
- If `LICENSE` exists, check the copyright year — if it says a previous year and commits span the current year, update to include the current year (e.g., `2025` → `2025-2026`)
- Check the copyright holder name matches the repo owner

## 3. Review with Technical Writer

Before applying fixes, launch a **Technical Writer** agent (`subagent_type: "Technical Writer"`) in parallel with your fix planning. Give it all the documentation files and the staleness findings from step 2. Tell it to:

- Review the accuracy and clarity of existing documentation
- Identify sections that are confusing, ambiguous, or poorly structured
- Suggest improved wording for any stale references you're about to fix
- Flag any missing documentation that should exist based on the codebase

Merge its suggestions into the fixes below. Do not change the author's voice or style — only incorporate suggestions that improve factual accuracy or fix genuinely confusing language.

## 4. Apply fixes

For each stale reference found:

- Use the **Edit** tool to surgically update only the stale content
- Preserve surrounding text, formatting, and tone exactly as-is
- Do NOT restructure sections, add new sections, or change wording that isn't factually wrong

**Specific fix patterns:**

| Staleness type | Fix approach |
|---|---|
| Dead file path | Update to new path, or remove reference if file was deleted |
| Wrong version | Update to current version from manifest |
| Missing feature in docs | Add a brief entry following the existing style/format |
| Removed feature in docs | Remove or mark as deprecated, following existing patterns |
| Checkbox mismatch | Toggle the checkbox to match issue state |
| Wrong command | Update to match current package manifest scripts |
| Stale directory listing | Update to match current structure |
| todo.txt completed item | Check the box `[x]`; if all done, suggest deleting the file |
| LICENSE stale year | Update copyright year range to include current year |

### CHANGELOG generation (only with `changelog` argument)

If `$ARGUMENTS` contains `changelog`:

1. Find the last CHANGELOG entry's date or version
2. Run `git log --oneline --after=<date>` to get commits since then
3. Group commits by type based on prefixes or content:
   - **Added** — new features (feat:, add:)
   - **Changed** — modifications to existing features
   - **Fixed** — bug fixes (fix:)
   - **Removed** — removed features
4. Write a new version entry at the top of CHANGELOG.md following the existing format
5. If no CHANGELOG.md exists, create one with the standard Keep a Changelog format

## 5. Commit changes

If any tracked files were modified:

```bash
git add <modified-doc-files>
git commit -m "$(cat <<'EOF'
Sync documentation with current codebase state

Updated stale references in project docs to match actual code.

Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>
EOF
)"
```

**Note:** `todo.txt` is untracked (in `.gitignore`) — edit it in place but do not `git add` it.

## Style guidelines
- Follow the standard output format in `_output-format.md`
- This skill is about accuracy, not creativity — never embellish or rewrite
- When in doubt, leave text unchanged — false positives (unnecessary edits) are worse than false negatives (missed staleness)
- Respect the author's formatting: if they use `##` headers, don't switch to `###`. If they use `-` for lists, don't use `*`
- Preserve blank lines, trailing newlines, and whitespace conventions
